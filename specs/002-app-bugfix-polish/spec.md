# Feature Specification: 应用问题修复与优化

**Feature Branch**: `002-app-bugfix-polish`
**Created**: 2026-03-29
**Status**: Clarified
**Input**: User description: "修复专注力训练应用的多项问题：主题切换不生效、主题默认跟随系统、全中文本地化、移除"查看全部"、修复问候语和数据真实性、用户信息可编辑、优化小游戏交互和导航、修复小游戏功能bug、删减游戏模块至3个、游戏规则弹窗机制、统一分数满分100、统计改为次数制"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 主题切换与系统跟随 (Priority: P1)

作为用户，我希望主题能够正确切换，并且默认跟随系统设置，这样我可以在不同环境下获得舒适的视觉体验。

**Why this priority**: 主题切换是用户个性化体验的基础功能，直接影响使用舒适度。

**Independent Test**: 可以独立测试主题切换功能，验证浅色/深色模式是否正确应用，以及系统主题变更时应用是否自动同步。

**Acceptance Scenarios**:

1. **Given** 用户首次打开应用，**When** 系统当前为深色模式，**Then** 应用自动以深色模式显示
2. **Given** 用户首次打开应用，**When** 系统当前为浅色模式，**Then** 应用自动以浅色模式显示
3. **Given** 用户在设置中选择手动切换主题，**When** 点击深色/浅色切换，**Then** 应用主题立即更新且保存用户选择
4. **Given** 用户已手动设置主题，**When** 重新打开应用，**Then** 应用显示用户上次选择的主题

---

### User Story 2 - 全中文本地化 (Priority: P1)

作为中文用户，我希望应用内所有文本都是中文，这样我可以无障碍地使用应用。

**Why this priority**: 语言本地化是用户体验的基本要求，英文文本会影响中文用户的使用体验。

**Independent Test**: 可以独立检查应用内所有可见文本，确保没有英文残留。

**Acceptance Scenarios**:

1. **Given** 用户浏览应用各个页面，**When** 查看按钮、标签、提示等文本，**Then** 所有文本均为中文
2. **Given** 用户查看导航栏，**When** 查看各个导航项，**Then** 导航文本均为中文
3. **Given** 用户查看设置页面，**When** 查看所有设置选项，**Then** 设置项文本均为中文
4. **Given** 用户查看错误提示，**When** 触发错误场景，**Then** 错误提示信息为中文

---

### User Story 3 - 首页优化与真实数据 (Priority: P1)

作为用户，我希望首页显示真实的训练数据和恰当的问候语，这样我可以准确了解自己的训练情况。

**Why this priority**: 真实数据是应用核心价值所在，虚假数据会降低用户信任。

**Independent Test**: 可以独立验证首页数据展示是否正确反映用户的实际训练记录。

**Acceptance Scenarios**:

1. **Given** 用户打开首页，**When** 根据当前时间，**Then** 显示符合时间段的问候语（如"早上好"、"下午好"、"晚上好"）
2. **Given** 首页显示训练模式区域，**When** 用户查看该区域，**Then** 不显示"查看全部"按钮
3. **Given** 用户有训练记录，**When** 查看首页训练洞察区域，**Then** 显示基于真实数据的统计（连续训练天数、总训练次数、平均专注度等）
4. **Given** 用户没有训练记录，**When** 查看首页训练洞察区域，**Then** 显示友好的空状态提示，引导用户开始训练

---

### User Story 4 - 用户资料编辑 (Priority: P2)

作为用户，我希望能够修改自己的名字和头像，这样我可以个性化我的账户信息。

**Why this priority**: 用户资料编辑是基本账户功能，提升用户归属感和个性化体验。

**Independent Test**: 可以独立测试用户资料编辑功能，验证修改后数据是否正确保存和显示。

**Acceptance Scenarios**:

1. **Given** 用户在个人资料页面，**When** 点击编辑名字，**Then** 可以输入新名字并保存
2. **Given** 用户修改了名字，**When** 保存成功后，**Then** 首页和其他页面显示新名字
3. **Given** 用户在个人资料页面，**When** 点击修改头像，**Then** 可以从预设头像中选择或上传自定义头像
4. **Given** 用户修改了头像，**When** 保存成功后，**Then** 首页和其他页面显示新头像

---

### User Story 5 - 小游戏交互优化 (Priority: P1)

作为用户，我希望在小游戏过程中能够随时返回或停止，并且界面简洁无干扰，这样我可以专注于训练。

**Why this priority**: 小游戏是核心训练功能，良好的交互体验直接影响训练效果。

**Independent Test**: 可以独立测试小游戏界面的导航隐藏和退出功能。

**Acceptance Scenarios**:

1. **Given** 用户进入小游戏，**When** 游戏进行中，**Then** 顶部和底部导航栏自动隐藏
2. **Given** 用户在游戏进行中，**When** 点击返回按钮或手势，**Then** 弹出确认对话框，确认后退出游戏
3. **Given** 用户在游戏进行中，**When** 点击停止按钮，**Then** 游戏暂停并显示选项：继续、重新开始、退出
4. **Given** 用户完成游戏或主动退出，**When** 返回游戏列表页面，**Then** 导航栏正常显示

---

### User Story 6 - 小游戏功能修复 (Priority: P1)

作为用户，我希望小游戏能够正常运行，没有功能bug，这样我可以顺利进行专注力训练。

**Why this priority**: 功能稳定性是用户体验的基础，bug会严重影响训练效果和用户信任。

**Independent Test**: 可以独立测试每个小游戏的完整流程，确保没有功能缺陷。

**Acceptance Scenarios**:

1. **Given** 用户开始任意小游戏，**When** 按规则进行操作，**Then** 游戏逻辑正确执行，得分准确计算
2. **Given** 用户完成游戏，**When** 达到结束条件，**Then** 正确显示成绩和统计信息
3. **Given** 游戏过程中出现异常，**When** 错误发生时，**Then** 应用不崩溃，显示友好的错误提示
4. **Given** 用户多次进行游戏，**When** 每次游戏结束，**Then** 成绩正确保存到训练记录

---

### User Story 7 - 游戏精简与规则弹窗 (Priority: P1)

作为用户，我希望只保留核心训练游戏（舒尔特表、字色干扰、序列记忆），并且在进入游戏时通过弹窗了解规则，这样我可以专注于最有价值的训练项目且不被复杂界面干扰。

**Why this priority**: 游戏精简降低认知负担，规则弹窗机制提升首次体验引导质量。

**Independent Test**: 可以独立验证游戏列表是否只包含 3 个游戏、规则弹窗是否按预期展示/隐藏、"不再提示"勾选是否持久生效。

**Acceptance Scenarios**:

1. **Given** 用户查看首页训练模式，**When** 浏览游戏卡片，**Then** 只显示舒尔特表、字色干扰、序列记忆 3 个游戏
2. **Given** 用户首次点击游戏卡片进入游戏页，**When** 页面加载完成，**Then** 自动弹出游戏规则弹窗
3. **Given** 规则弹窗底部有"不再提示"勾选框，**When** 用户勾选后关闭弹窗，**Then** 下次进入该游戏不再自动弹出规则
4. **Given** 用户未勾选"不再提示"，**When** 再次进入同一游戏，**Then** 规则弹窗仍然自动展示
5. **Given** 用户在游戏开始页面，**When** 点击"展示规则"按钮，**Then** 再次弹出该游戏的规则弹窗
6. **Given** 已删除游戏（听觉注意、分类逻辑、情景联想）的历史记录，**When** 执行数据清理，**Then** 这些游戏的所有训练记录被完全删除

---

### User Story 8 - 分数统一与次数制统计 (Priority: P1)

作为用户，我希望所有游戏分数统一为满分 100，训练统计以"次数"而非"时间"为单位，每日目标以训练次数衡量，这样我可以更直观地理解自己的训练表现。

**Why this priority**: 统一的评分和统计体系是用户理解训练成果的基础。

**Independent Test**: 可以独立验证分数是否在 0-100 范围、统计页面是否显示次数而非分钟、每日目标是否为次数。

**Acceptance Scenarios**:

1. **Given** 用户完成任意游戏，**When** 查看得分，**Then** 分数在 0-100 范围内，100 为满分
2. **Given** 用户查看训练统计页面，**When** 查看各项指标，**Then** 所有统计以"训练次数"为单位展示，不显示时间维度
3. **Given** 用户查看每日目标，**When** 查看进度，**Then** 目标显示为"N次训练"而非"N分钟"
4. **Given** 用户设置每日目标，**When** 选择目标值，**Then** 可选次数选项（如 3次/5次/10次）
5. **Given** 数据库中存在历史训练记录，**When** 执行分数标准化，**Then** 所有历史分数被清除，从新标准开始

---

### Edge Cases

- 主题切换时系统主题同时改变，应用如何处理冲突？
- 用户手动设置主题后，是否还应该响应系统主题变化？
- 训练数据为空时，如何优雅地展示首页洞察区域？
- 用户在游戏中意外退出应用（如接打电话），如何恢复状态？
- 用户快速连续点击游戏按钮，是否会导致重复启动？
- 游戏过程中设备旋转，界面是否正常适配？
- 已删除游戏的历史记录清理失败时，是否影响应用正常使用？
- 分数恰好为 0 或 100 的边界情况，显示是否正确？
- 用户清除浏览器数据后，"不再提示"偏好被重置，规则弹窗应重新自动展示
- 首页游戏卡片从 6 个变为 3 个后的布局适配（2列网格可能出现不对称）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须支持浅色和深色两种主题模式
- **FR-002**: 系统默认必须跟随操作系统主题设置
- **FR-003**: 用户必须能够在设置中手动选择主题模式
- **FR-004**: 用户手动选择的主题偏好必须持久化保存
- **FR-005**: 应用内所有可见文本必须是中文，不得出现英文
- **FR-006**: 首页问候语必须根据当前时间段动态显示（早上/下午/晚上）
- **FR-007**: 首页训练模式区域不得显示"查看全部"按钮
- **FR-008**: 首页训练洞察数据必须基于用户的真实训练记录计算
- **FR-009**: 当没有训练记录时，必须显示友好的空状态引导
- **FR-010**: 用户必须能够在个人资料页面修改显示名称
- **FR-011**: 用户必须能够从预设头像中选择或上传自定义头像
- **FR-012**: 修改后的用户信息必须立即在所有相关页面生效
- **FR-013**: 小游戏运行时顶部和底部导航栏必须自动隐藏
- **FR-014**: 用户必须能够在游戏过程中暂停或退出游戏
- **FR-015**: 退出游戏前必须显示确认对话框防止误操作
- **FR-016**: 所有小游戏必须逻辑正确，无功能缺陷
- **FR-017**: 游戏成绩必须正确计算并保存到训练记录
- **FR-018**: 应用只保留 3 个训练游戏：舒尔特表(schulte)、字色干扰(stroop)、序列记忆(sequence)
- **FR-019**: 必须删除听觉注意(auditory)、分类逻辑(classify)、情景联想(story)的游戏代码、路由和所有引用
- **FR-020**: 已删除游戏的所有历史训练记录必须从数据库中清除
- **FR-021**: 用户首次进入某游戏时必须自动弹出该游戏的规则说明弹窗
- **FR-022**: 规则弹窗底部必须提供"不再提示"勾选框，勾选后该游戏不再自动弹出规则
- **FR-023**: "不再提示"偏好必须按游戏独立持久化存储
- **FR-024**: 游戏开始页面必须始终提供"展示规则"按钮，用户可随时重新查看规则
- **FR-025**: 点击游戏卡片后进入游戏开始页面，展示：开始按钮、简短描述文字、展示规则按钮
- **FR-026**: 所有游戏分数必须统一为 0-100 分制，100 为满分
- **FR-027**: 迁移到新评分制时，所有历史分数必须被清除
- **FR-028**: 训练统计必须以"训练次数"为单位，不得使用时间维度
- **FR-029**: 每日训练目标必须从"分钟"改为"训练次数"（如 3次/5次/10次）
- **FR-030**: 设置页的每日目标选项必须改为次数选项

### Key Entities *(include if feature involves data)*

- **用户偏好 (UserPreference)**: 存储用户主题设置（自动/浅色/深色）、每游戏规则提示偏好（不再提示标记）、每日目标次数
- **用户信息 (UserProfile)**: 用户显示名称、头像URL、创建时间等
- **训练记录 (TrainingRecord)**: 每次训练的类型、分数(0-100)、时间戳等；仅包含 schulte/stroop/sequence 三种模式
- **训练统计 (TrainingStats)**: 累计训练天数、总次数、平均分数等聚合数据（以次数为维度，不含时间维度）
- **游戏配置 (GameConfig)**: 仅 3 个游戏的配置参数、规则定义和描述文案

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 主题切换响应时间小于200毫秒，切换过程无闪烁
- **SC-002**: 首次打开应用时，95%以上的用户看到与系统一致的主题
- **SC-003**: 应用内英文文本出现率为0%，所有用户可见文本均为中文
- **SC-004**: 首页训练洞察数据与真实数据的匹配度为100%
- **SC-005**: 用户信息修改后，在3秒内同步到所有相关页面
- **SC-006**: 小游戏加载时间小于2秒，游戏过程中帧率保持在30fps以上
- **SC-007**: 游戏过程中导航栏隐藏率为100%，无任何导航元素干扰
- **SC-008**: 游戏功能正确率100%，无已知功能缺陷
- **SC-009**: 训练记录保存成功率99.9%以上，无数据丢失
- **SC-010**: 用户完成任意操作后，应用崩溃率低于0.1%
- **SC-011**: 应用中不存在已删除游戏(auditory/classify/story)的任何代码引用或路由
- **SC-012**: 所有游戏得分严格在 0-100 范围内，无超范围分数
- **SC-013**: 统计页面所有指标以"次数"为单位，不出现"分钟"等时间单位
- **SC-014**: "不再提示"偏好持久化，关闭浏览器后重新打开仍生效
- **SC-015**: 每日目标显示为"N次"，设置页提供次数选项

## Clarifications

### Session 2026-05-28

- Q: 已删除的 3 个游戏（听觉注意、分类逻辑、情景联想）的历史训练记录如何处理？ → A: 完全删除这 3 个游戏的所有历史记录数据。
- Q: "分数满分全部为100" — 已有数据库中的历史分数如何处理？ → A: 全部重置：清除所有历史分数，统一从新标准开始。
- Q: 统计改为以"次数"为单位后，每日训练目标是否也相应从"分钟"改为"训练次数"？ → A: 改为次数制：每日目标变为"N次训练"。
- Q: 用户勾选"不再提示"游戏规则后，是否需要提供重新开启的入口？ → A: 在每个游戏的开始页面始终保留"展示规则"按钮，无需额外重置功能。

## Assumptions

- 用户使用现代浏览器或移动设备，支持CSS变量和媒体查询
- 系统主题变化事件能够被应用正常监听
- 用户的训练数据存储在本地IndexedDB中，无需后端同步
- 头像上传功能限制文件类型为图片，大小不超过2MB
- 小游戏支持离线运行，不依赖网络连接
- 用户对专注力训练应用有基本的使用预期和理解
- 已删除 3 个游戏的数据清除操作为一次性迁移，不保留回滚能力
- "不再提示"偏好存储在本地（IndexedDB 或 localStorage），不同设备间不同步
- 每日目标次数选项的具体数值（如 3/5/10）可在实现阶段确定
